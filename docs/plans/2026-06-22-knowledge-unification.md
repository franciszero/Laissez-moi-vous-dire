# 设计文稿：知识点系统统一（陈述+题目一族 / 折叠 AI 精练 / 全卡历史）

最后更新：2026-06-22。面向接手 agent 与外审。配套：`HANDOFF.md §3`、skill `llm-graded-practice`。

## 0. 目标（用户三点诉求）
1. **陈述与题目是一族的两支**：创建知识点时，规则陈述必须配套应用题，不允许只入"裸陈述"。
2. **折叠 AI 精练进知识点**：去掉独立的「🤖 AI 精练」button（用户不知道里面有什么）。复习完一条知识点 → 紧接着做它的题；机判当场判、自由产出题由本地 LLM 判。模型状态用**页面顶部一条浮动状态条**显示（加载中/已加载/卸载中/已卸载）。模型按需加载、闲置自动卸载。
3. **每个知识点都有历史**：能看到"上次我怎么答的、错在哪"，强化记忆。

## 1. 现状（事实，别重推导）
- 两个**故意分开**的独立流程：`cp_active`（知识点卡复习，机判 check_fr / 自评）与 `llm_active`（AI 精练，5 道固定产出题，本地模型批改）。两者都"绝不碰 pool/current_word"。
- 知识点卡 SRS 状态存 `checkpoints` 表（card_id 主键：correct_streak/interval_days/due_at）。**没有逐次作答历史**。
- 仅 AI 精练存逐次历史：`ai_attempts` 表（cue_id/answer/verdict/feedback(spans)/hint_level/...）。
- 本地模型：35B-A3B，加载 10–12s、+~1.4GB swap、卸载后内存回落约 22GB；生命周期在 `llm.py`（按需加载/退出/切换/闲置5min/atexit 卸载），**绝不写 Hermes 配置**。
- 知识点卡 = species 卡 + checkpoint_groups 的 practice_cards；机判卡有 answer、自评卡 answer=null。
- **更正（外审发现）**：两个流程**只在 `pool/current_word` 层面分开**；在 SRS 数据层**早已耦合**——AI 精练的 5 道题已 `ensure_checkpoint`(app.py:1112)、判定后 `update_checkpoint`(app.py:2157–2159)，即 **AI cue 已是 `checkpoints` 表里的行、共用 `srs.next_schedule`**。所以「折叠」在**数据层基本已完成**，Phase C 的真实工作是 **UI/生命周期统一，不是数据合并**。

## 2. 设计（分三期，每期独立可上线）

### Phase A（先做）：全知识点逐次历史
- 新表 `checkpoint_attempts(id, card_id, created_at, user_answer, correct)`，幂等建表（沿用 `_ensure_*_table` 模式）。**去掉 verdict（可由 correct 导出）和 hint_level（只属 LLM，归 `ai_attempts`）**。
- 只在**机判判定后（app.py 机判 site，`ans` 在作用域）**和**自评"我对/我错"后（self site，`user_answer=NULL`）**写历史。**绝不在 AI 精练判定处写**（那条已进 `ai_attempts`，否则双写）。**不改 SRS 算法**。
- 不变量（带测试）：机判/自评卡作答 → `checkpoint_attempts` 有行；**AI cue 作答 → 只进 `ai_attempts`，不进 `checkpoint_attempts`**。
- 卡片复习界面：展示该卡近 N 次（机判："上次你写了 X，对/错"；自评："上次：我对/我错 · 日期"）。
- 验收：上述不变量成立；卡片视图能看到上次；老测试不破。

### Phase B：规范 "陈述必配题"
- 约定（写进 `llm-graded-practice` 或卡片方法论 skill）：每个知识点 = 1 条陈述 + ≥1 道应用题。应用题形式按内容择一：
  - 形/变位/搭配 → 机判填空；选择/辨析 → 机判决策；自由产出 → AI 判；纯过程/策略（无唯一答案）→ 自评应用题。
- **可执行化（外审发现：否则会退化成"造假题"或"全豁免"）**：给卡加字段 `application:{kind: machine|self|llm}` 或 `exempt:<reason>`；扩展 `scripts/coverage_report.py`，当某 species 既无 application 又无 exempt 时**报错**。这样"陈述必配题"是闸门不是口号。
- 落地：补齐现有"裸陈述"卡的配套题（用上面的覆盖闸扫出缺口）。
- 验收：方法论 skill 有该条；coverage 闸通过（每条要么有题要么有豁免理由）。

### Phase C（最后，最大）：折叠 AI 精练 + 顶部状态条 + 懒加载
- 把 5 道 AI 产出题**作为"产出题"卡并入知识点组**（按主题归到相关 group，可被搜索/发现），不再用独立 button。
- **模型生命周期改为懒加载 + 黏着 + 闲置卸载**（关键设计决定，区别于用户初版"进块即加载"）：
  - 进入含产出题的知识块**不加载**模型；
  - **用户真正去做第一道产出题（提交）时才加载**（顶部状态条显示"加载中…"→"已加载"）；
  - 加载后**保持热**：用户可连续做当前所有待做产出题（跨主题）不再重复加载；
  - **闲置超时 / 离开知识区**时卸载（状态条显示"卸载中…"→"已卸载"）。
  - 理由：加载一次 10–12s+swap；"进块即加载"会让复习中反复进出反复卡 10–12s，反而不连贯。大多数卡是机判/自评，根本不触发模型。
- **闲置卸载必须提升到复习引擎级（外审发现）**：现 `_llm_idle_watch` fragment 只在 AI 视图渲染时跑；折叠后用户多数时间在机判/自评卡上，若 fragment 仍绑在产出题子视图 → **模型永不卸载、常驻 47GB**。所以 idle-unload fragment 必须在**知识区每次 rerun 都跑**（不论当前是哪类卡），它是**唯一保证的兜底**；显式 exit 卸载只是 best-effort。**最大泄漏窗口 = 闲置阈值**。
- **加载并发护栏（外审发现）**：`llm.load()` 同步阻塞 ~10–12s，期间 fragment tick / 二次提交可能重入。加单一 `llm_load_in_progress` 标志，保证加载窗口内不会再调 `llm.load()`。
- **id 稳定（外审发现）**：把 5 道 `LLM_EXERCISES`（`L22:ai:*`）迁进 manifest/group 时 **id 必须逐字保留**，否则现有 SRS 状态 + `ai_attempts` 历史会成孤儿。
- 顶部浮动状态条：单一来源反映 `llm.is_loaded()` + 会话状态；**与 idle-watch 共用同一个 fragment/状态**，别开两个轮询。
- 合并两流程时**保留状态隔离纪律**：产出题判分仍走 `aigrade`（人工规范 + 代码确定性染色 + 三态终判），不退回"模型当裁判"。
- 验收：无独立 AI button；产出题在知识点组里可见且 id 不变；状态条正确反映 加载/卸载；**AppTest 覆盖"提交产出题→加载一次；之后在机判卡上闲置→自动卸载"**；产出题仍按 `llm-graded-practice` 判。

## 3. 风险与缓解
- **加载抖动**：懒加载+黏着+闲置卸载（见 Phase C），不"进块即加载"。
- **两流程合并的状态纠缠**：`cp_*` 与 `llm_*` 合并后仍保持单一活动视图、清晰的进入/离开清理；先小步（产出题作为一种卡类型嵌入），不重写整个复习引擎。
- **"陈述必配题"边界**：纯过程/策略类允许"自评应用题"作为题，不强行造机判。
- **资源研判未实测**：模型真实质量仍由离线 `scripts/llm_grade_bakeoff.py` 决定，本设计不假设模型已可信。

## 4. 明确不做
- 不动 SRS 间隔算法、不动 matcher。
- 不建跨课共享层（HANDOFF §5）。
- 不把模型判定当 SRS 真相（仍人工三态终判）。
- Phase C 不追求"每张卡 0 延迟"——首道产出题的 10–12s 加载是硬成本，只做到不散落、不重复。

## 5. 顺序与理由
A（历史，独立、价值即时）→ B（规范，便宜）→ C（折叠，最大、交互最需想清楚）。每期能单独 ship、单独验证。

## Phase D（来自 product-story-critique 评审）：侧栏 8→3，卡即数据分派
评审结论：8 个学习入口本质是 2 族（练词 / 练卡），被按"内部实现轴"切碎；"到期"碎成两处。
目标侧栏：**学这课 / 今天复习 / 知识点（含所有卡型）**；错词→过滤、变形→模式（降级，不删功能）。
保留（别丢）：确定性掌握真相 vs LLM 陪练、species→manifest 卡即数据管线、按需加载/卸载纪律。
不做跨课共享（HANDOFF §5：等 3+ 课真痛）。每期 设计→开发→验收→commit/push。

- **D1 ✅**：修模式互斥（`_leave_overlays` 统一清 cp/conj，修"词入口被卡模式遮挡"的潜伏 bug）；统一「今天复习」banner（词 + 知识点卡一处可见、各一键）。
- **D2 ✅**：`render_card` 按 `kind` 分派；动词变位卡并入知识点 deck（D2a 加入 + D2b 撤独立按钮/流程）；checkpoint/自评/变位三类卡共一套复习流程与导航。L22 deck 194→215→221。
- **D3（暂缓，留给用户在场时做）**：合并 AI 精练入 deck。分派基础设施已就绪（D2），但其正确性取决于**模型生命周期**（首张 AI 卡才加载、在别的卡上闲置时卸载、不泄漏 47GB——评审 MUST-FIX）。该行为**无法自动验证**（AppTest 把模型 mock 了，真 35B 加载只能在用户机上验），无人盯着上线风险太高。AI 入口现仍独立可用。
- **D4 ✅**：补不规则动词变位卡。**mlconjug3（用户所选）在本机坏了**——其打包模型需 sklearn 1.6.x、本机 1.9.0（unpickle 失败 `_loss`）；verbecc API 不配合。故未加脆弱依赖，改用**人工核对表 + 逐格测试验证**（用户的备选 B）：être/avoir/aller/voir/pouvoir/vouloir 6 个已收，规则动词仍规则生成。仍缺的不规则待补表或修好的库。
